# Важно для интервью по Configuration & Options

Разработчики уровня Senior должны понимать не только как прочитать строку из конфига, но и как устроена цепочка ответственности провайдеров и жизненный цикл объектов настроек.

## Ключевые вопросы и ответы

### 1. Чем отличается IOptions от IOptionsSnapshot и IOptionsMonitor?
Это классический вопрос.
- **IOptions:** Singleton. Читается один раз. Не обновляется.
- **IOptionsSnapshot:** Scoped. Обновляется при каждом HTTP-запросе. Идеален для веб-запросов.
- **IOptionsMonitor:** Singleton. Обновляется мгновенно при изменении файла. Имеет callback `OnChange`.

### 2. Как работают User Secrets?
**Ответ:** Это механизм для локальной разработки. Секреты хранятся в JSON файле в папке профиля пользователя (`%APPDATA%\Microsoft\UserSecrets\...`), а не в папке проекта. Это гарантирует, что секреты никогда не попадут в Git. В продакшене они заменяются переменными окружения или Key Vault.

### 3. В каком порядке применяются источники конфигурации?
**Ответ:** В порядке их регистрации. Последний зарегистрированный провайдер «побеждает». Обычно: JSON $\rightarrow$ Environment $\rightarrow$ CLI args. Если в JSON указано `Port=80`, а в CLI `Port=443`, итоговым будет 443.

### 4. Зачем нужен PostConfigure?
**Ответ:** Для выполнения действий после того, как все основные `Configure` были выполнены. Это полезно для задания значений по умолчанию, которые нельзя перебить через конфиг, или для вычисления зависимых параметров.

### 5. Как реализовать «безопасную» конфигурацию (Fail Fast)?
**Ответ:** Использовать `AddOptions().ValidateDataAnnotations().ValidateOnStart()`. Это позволяет приложению упасть в первую же секунду запуска с детальным описанием ошибки в логах, вместо того чтобы работать с некорректными данными и выдавать ошибки в рантайме.

## Сценарий: «Оптимизация обновления настроек»
**Задача:** «У нас есть Singleton-сервис (например, фоновый воркер), которому нужно менять лимит обработки сообщений без перезагрузки приложения».
**Правильный ответ:**
1. Внедрить `IOptionsMonitor<MySettings>` в Singleton-сервис.
2. Использовать `_monitor.CurrentValue` при каждой проверке лимита.
3. (Опционально) Подписаться на `_monitor.OnChange(...)`, чтобы обновить какие-то внутренние кэши сервиса при изменении конфига.
