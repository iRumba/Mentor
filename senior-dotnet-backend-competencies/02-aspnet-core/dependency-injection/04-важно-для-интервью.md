## Важно для интервью по DI

Раздел содержит наиболее каверзные вопросы и архитектурные ловушки, связанные с Dependency Injection в .NET.

### 1. Captive Dependency (Плененная зависимость)

Это ситуация, когда сервис с более длинным временем жизни (например, Singleton) зависит от сервиса с более коротким временем жизни (например, Scoped).

- **Проблема**: Scoped-сервис "пленяется" внутри Singleton и продолжает жить до конца работы приложения. Это приводит к:
    - Утечкам памяти.
    - Ошибкам в работе БД (например, `DbContext` не предназначен для долгой жизни и многопоточности).
    - Ошибкам обновления данных (старое состояние в кэше Scoped-сервиса).
- **Решение**: 
    - Использовать `IServiceScopeFactory` внутри Singleton (создавать scope вручную).
    - Пересмотреть архитектуру (возможно, сервис должен быть Transient).

### 2. Scope Validation

В режиме `Development` ASP.NET Core по умолчанию включает проверку scopes. Если вы попытаетесь внедрить Scoped-сервис в Singleton, приложение выбросит исключение при старте. 
В `Production` эта проверка отключена для производительности, поэтому Captive Dependency может проявиться как трудноуловимый баг.

### 3. TryAdd* методы

Методы `TryAddSingleton`, `TryAddScoped` и `TryAddTransient` (из `Microsoft.Extensions.DependencyInjection.Extensions`) регистрируют сервис только в том случае, если он еще не был зарегистрирован под данным интерфейсом.
- **Зачем это нужно?** Полезно при создании библиотек. Вы регистрируете "дефолтную" реализацию, но позволяете пользователю вашей библиотеки переопределить её, зарегистрировав свою реализацию до вызова вашего метода расширения.

### 5. Порядок регистрации

Порядок имеет значение: если один интерфейс зарегистрирован несколько раз:
- `GetService<T>()` вернет **последнюю** зарегистрированную реализацию.
- `GetServices<T>()` (возвращая `IEnumerable<T>`) вернет **все** зарегистрированные реализации в порядке их добавления.
