## Регистрация сервисов

Регистрация сервисов в ASP.NET Core выполняется через `IServiceCollection`, которая определяет, как контейнер должен создавать экземпляры зависимостей и как долго они должны жить.

### Время жизни сервисов (Service Lifetimes)

1. **Singleton (`AddSingleton<T>`)**
   - **Суть**: Один экземпляр создается при первом запросе или при запуске приложения и используется всеми последующими запросами до завершения работы процесса.
   - **Когда использовать**: Для кэшей, конфигураций, бесстейт-клиентов (например, `HttpClient` через `IHttpClientFactory`) или общих ресурсов.
   - **Риски**: Состояние внутри Singleton должно быть потокобезопасным.

2. **Scoped (`AddScoped<T>`)**
   - **Суть**: Новый экземпляр создается для каждого логического запроса (обычно один HTTP-запрос). В рамках одного запроса все компоненты получают один и тот же экземпляр.
   - **Когда использовать**: Для контекстов баз данных (`DbContext`), пользовательских сессий или сервисов, которые должны сохранять состояние в рамках одного запроса.
   - **Особенности**: В Minimal API и контроллерах scope создается автоматически на каждый запрос.

3. **Transient (`AddTransient<T>`)**
   - **Суть**: Новый экземпляр создается каждый раз, когда сервис запрашивается из контейнера.
   - **Когда использовать**: Для легких, бесстейт-сервисов, которые не сохраняют состояние между вызовами.

### Keyed Services (.NET 8+)
В .NET 8 появилась возможность регистрировать несколько реализаций одного интерфейса, различая их ключами:
```csharp
builder.Services.AddKeyedSingleton<IMessageService, EmailService>("email");
builder.Services.AddKeyedSingleton<IMessageService, SmsService>("sms");

// Использование
public class NotificationController([FromKeyedServices("email")] IMessageService emailService) { ... }
```
Это избавляет от необходимости создавать отдельные интерфейсы-маркеры или сложные фабрики вручную.

### Рекомендации и Best Practices
- **Избегайте Service Locator**: Не внедряйте `IServiceProvider` напрямую в бизнес-логику. Это скрывает зависимости и усложняет тестирование.
- **Принцип единственной ответственности**: Если сервис требует слишком много зависимостей в конструкторе, возможно, его стоит разделить.
- **Используйте интерфейсы**: Регистрация через `AddSingleton<IInterface, Implementation>()` облегчает мокирование в Unit-тестах.
