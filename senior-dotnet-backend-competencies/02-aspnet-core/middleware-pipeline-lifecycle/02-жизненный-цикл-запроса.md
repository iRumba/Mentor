## Жизненный цикл запроса

Понимание пути HTTP-запроса через ASP.NET Core критично для отладки производительности и проблем с безопасностью.

### Детальный путь запроса (Request Lifecycle)

1. **Сервер (Kestrel / HTTP.sys / IIS)**:
   - Принимает TCP-соединение.
   - Парсит HTTP-запрос в объект `HttpContext`.
   - Передает управление в ASP.NET Core Pipeline.

2. **Middleware Pipeline (Вход)**:
   - Запрос проходит через зарегистрированные middleware в порядке их объявления (LIFO - Last In First Out для выхода).
   - Примеры: `ExceptionHandler` $\rightarrow$ `HSTS` $\rightarrow$ `HTTPS Redirection` $\rightarrow$ `StaticFiles` $\rightarrow$ `Routing`.

3. **Routing (Маршрутизация)**:
   - `UseRouting()` сопоставляет URL запроса с конкретным эндпоинтом (Endpoint).
   - Информация об эндпоинте сохраняется в `HttpContext.GetEndpoint()`, но сам эндпоинт еще не выполняется.

4. **Authorization & Middleware после Routing**:
   - `UseAuthorization()` проверяет права доступа к выбранному эндпоинту.

5. **Endpoint Execution**:
   - `UseEndpoints()` (или implicit в Minimal API) вызывает код контроллера или лямбда-выражение.
   - Здесь работают MVC-фильтры (Action Filters, Resource Filters).

6. **Middleware Pipeline (Выход)**:
   - Ответ возвращается вверх по цепочке middleware.
   - Логика, написанная после `await next(context)`, выполняется именно здесь.

### Схема потока данных
`Запрос` $\rightarrow$ `Kestrel` $\rightarrow$ `Middleware 1` $\rightarrow$ `Middleware 2` $\rightarrow$ `Routing` $\rightarrow$ `Auth` $\rightarrow$ `Controller` $\rightarrow$ `Auth` $\rightarrow$ `Middleware 2` $\rightarrow$ `Middleware 1` $\rightarrow$ `Kestrel` $\rightarrow$ `Ответ`

### Общие ошибки
- **Модификация Response после начала записи**: Попытка изменить заголовки ответа после того, как тело ответа начало отправляться клиенту (`Response.HasStarted == true`), приведет к исключению.
- **Игнорирование порядка**: Попытка использовать `HttpContext.GetEndpoint()` до вызова `app.UseRouting()` вернет `null`.

### Сравнение: ASP.NET Core vs Legacy ASP.NET
В legacy ASP.NET (Web Forms/MVC 5) использовались `HttpModules` и `HttpHandlers`. В ASP.NET Core всё заменено единым механизмом Middleware, что значительно упростило архитектуру и улучшило производительность (отказ от тяжелого `System.Web`).
