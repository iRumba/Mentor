# Scaling (Масштабирование SignalR)

По умолчанию SignalR хранит информацию о соединениях и группах в памяти одного сервера. В распределенной системе с несколькими экземплярами приложения (Server Farm) возникает проблема: если Клиент А подключен к Серверу 1, а Сервер 2 хочет отправить ему сообщение, он не сможет этого сделать, так как не «знает» о соединении Клиента А.

Для решения этой проблемы используется механизм **Backplane**.

## Как работает Backplane

Backplane выступает в роли шины сообщений. Когда Сервер 1 отправляет сообщение группе или пользователю, он отправляет это сообщение в Backplane. Все остальные серверы (Сервер 2, Сервер 3 и т.д.) подписаны на этот Backplane, получают сообщение и пересылают его своим локально подключенным клиентам.

### Варианты реализации Backplane:

#### 1. Redis Backplane (`Microsoft.AspNetCore.SignalR.StackExchangeRedis`)
Самый популярный вариант для self-hosted решений. Использует механизм Pub/Sub в Redis.
- **Плюсы:** Высокая производительность, доступность, независимость от облака.
- **Минусы:** Требуется управление своим Redis-сервером. При очень больших нагрузках Redis может стать узким местом.

#### 2. Azure SignalR Service
Полноценный управляемый сервис (Managed Service). В этом случае клиенты подключаются не к вашему серверу, а напрямую к Azure SignalR Service. Ваш сервер общается с этим сервисом через SDK.
- **Плюсы:** Снимает нагрузку с ваших серверов (WebSocket-соединения Termination происходит в Azure), автоматическое масштабирование, высокая доступность.
- **Минусы:** Зависимость от Azure, стоимость.

## Сравнение архитектур

| Характеристика | Default (Single Server) | Redis Backplane | Azure SignalR Service |
| :--- | :--- | :--- | :--- |
| **Масштабируемость** | Только вертикальная | Горизонтальная | Высшая (Managed) |
| **Нагрузка на CPU/RAM** | Высокая (держать WS) | Высокая (держать WS) | Низкая (делегировано) |
| **Сложность настройки** | Нулевая | Низкая (конфиг Redis) | Средняя (Azure Setup) |
| **Надежность** | Одно точка отказа | Зависит от Redis | Очень высокая |

## Важные нюансы при масштабировании

### 1. Sticky Sessions (Липкие сессии)
Если вы используете Redis или любой другой Backplane, но **не** используете Azure SignalR Service, ваш Load Balancer **обязательно** должен поддерживать Sticky Sessions (Affinity).
**Почему?** Потому что процесс Negotiation (согласование транспорта) и само установление соединения должны происходить с одним и тем же экземпляром сервера. Если первый запрос ушел на Сервер 1, а второй (установление WS) на Сервер 2, соединение будет разорвано.

### 2. Производительность и Задержки
Использование Backplane добавляет дополнительный прыжок (hop) для каждого сообщения. 
- Сообщение $\rightarrow$ Сервер 1 $\rightarrow$ Redis $\rightarrow$ Сервер 2 $\rightarrow$ Клиент.
В большинстве случаев эта задержка незаметна, но для критически быстрых систем (например, высокочастотный трейдинг) это может быть проблемой.

### 3. Очистка ресурсов
При использовании Redis убедитесь, что настроен правильный TTL для ключей, чтобы избежать утечек памяти в случае некорректного завершения работы узлов сервера.
