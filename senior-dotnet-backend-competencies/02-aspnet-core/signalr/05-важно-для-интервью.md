# Важно для интервью по SignalR

На позицию Senior .NET Backend разработчика часто задают вопросы, проверяющие понимание внутренностей SignalR, а не просто умение вызвать `SendAsync`.

## Ключевые темы и ответы

### 1. Процесс Negotiation (Согласование)
**Вопрос:** Что происходит, когда клиент вызывает `.start()`?
**Ответ:** Клиент отправляет HTTP POST запрос на `/negotiate`. Сервер отвечает списком доступных транспортов (WebSockets, SSE, Long Polling) и, если применимо, идентификатором соединения (`connectionId`). После этого клиент выбирает лучший транспорт из поддерживаемых и пытается установить соединение.

### 2. Жизненный цикл и State Management
**Вопрос:** Можно ли хранить данные в полях класса Hub?
**Ответ:** Нет. Hub создается заново на каждый вызов метода. Для хранения состояния нужно использовать внешние хранилища (Redis, БД) или `IHttpContextAccessor` для доступа к данным текущего запроса.

### 3. Разница между `User` и `ConnectionId`
**Вопрос:** В чем разница при отправке сообщения пользователю и конкретному соединению?
**Ответ:** 
- `Clients.Client(connectionId)` отправит сообщение в одну конкретную вкладку/сессию.
- `Clients.User(userId)` отправит сообщение **всем** активным соединениям этого пользователя (например, в браузер на ПК и в приложение на телефоне одновременно). Для этого должен быть настроен `IUserIdProvider`.

### 4. Strongly-typed Hubs
**Вопрос:** Зачем нужны типизированные хабы?
**Ответ:** Чтобы избежать «магических строк» в именах методов. Это позволяет использовать интерфейс на стороне сервера, обеспечивая проверку типов при компиляции и упрощая рефакторинг.

### 5. Интеграция с фоновыми задачами
**Вопрос:** Как отправить сообщение клиенту из `IHostedService` (Background Worker), если у нас нет экземпляра Hub?
**Ответ:** Нужно внедрить `IHubContext<T>` в конструктор фонового сервиса. `IHubContext` предоставляет доступ к API рассылки сообщений (`Clients.All`, `Clients.User` и т.д.) вне контекста активного вызова метода хаба.

## Сценарии-задачи (Case Studies)

**Задача:** «Нам нужно реализовать систему уведомлений о статусе заказа. Заказ меняется в БД фоновым процессом, и мы должны уведомить пользователя в реальном времени».
**Правильный ответ:**
1. Создать `OrderHub : Hub`.
2. В фоновом сервисе внедрить `IHubContext<OrderHub>`.
3. Использовать `_hubContext.Clients.User(userId).SendAsync("OrderUpdated", orderData)`.
4. Важно упомянуть использование Azure SignalR Service или Redis Backplane, если приложение будет масштабироваться горизонтально.

**Задача:** «Как обеспечить безопасность в SignalR, чтобы пользователь не мог вступить в чужую группу (например, чат другого отдела)?»
**Правильный ответ:**
В методе `JoinGroup(string groupName)`, который вызывает клиент, добавить проверку прав доступа. Использовать `Context.User` для получения claims пользователя и проверить в БД/сервисе прав, имеет ли данный пользователь доступ к указанному `groupName`. Только после успешной проверки вызывать `Groups.AddToGroupAsync`.
