# Что такое SignalR

SignalR — это open-source библиотека для ASP.NET Core, которая упрощает добавление функций реального времени в веб-приложения. Вместо того чтобы клиент постоянно опрашивал сервер (polling), сервер сам «толкает» (push) данные клиенту.

## Технические внутренности и Транспорты

SignalR не изобретает новый протокол, а предоставляет абстракцию над существующими. При установке соединения происходит процесс **Negotiation** (согласование), в ходе которого клиент и сервер определяют наиболее эффективный транспорт:

1. **WebSockets**: Самый современный и эффективный транспорт. Обеспечивает полноценный полнодуплексный (full-duplex) обмен данными через одно TCP-соединение. Требует поддержки на стороне сервера, клиента и прокси-серверов.
2. **Server-Sent Events (SSE)**: Позволяет серверу отправлять данные клиенту через HTTP. Работает только в одну сторону (сервер -> клиент).
3. **Long Polling**: «Запасной» вариант. Клиент делает запрос, сервер держит его открытым до появления новых данных или тайм-аута. После получения ответа клиент тут же открывает новый запрос.

**Порядок выбора (Fallback):** WebSockets $\rightarrow$ SSE $\rightarrow$ Long Polling.

## Преимущества и Best Practices

- **Автоматическое управление:** Вам не нужно писать код для обработки переключений между транспортами или переподключений.
- **Hubs:** Использование концепции «хаба» позволяет структурировать логику общения как обычные методы C#, которые вызываются из JS/TS.
- **Производительность:** Для высоконагруженных систем рекомендуется использовать бинарный протокол **MessagePack** вместо JSON для уменьшения размера пакетов.

## Подводные камни (Pitfalls)

- **State Management:** Соединения в SignalR по умолчанию являются эфемерными. Не полагайтесь на хранение состояния внутри экземпляра Hub, так как Hub создается заново на каждый вызов. Используйте внешние хранилища (Redis, БД).
- **Proxy/Load Balancer:** При использовании WebSockets убедитесь, что ваш прокси (Nginx, IIS, Azure App Service) поддерживает их и настроен на корректную передачу заголовков `Upgrade`.
- **CORS:** SignalR требует четкой настройки CORS, особенно если используется механизм Sticky Sessions.

## Сравнение с альтернативами

| Характеристика | SignalR | Чистые WebSockets | gRPC |
| :--- | :--- | :--- | :--- |
| **Сложность внедрения** | Низкая (из коробки) | Средняя (нужен ручной парсинг) | Средняя/Высокая |
| **Транспорты** | WS, SSE, Long Polling | Только WS | HTTP/2 |
| **Формат данных** | JSON / MessagePack | Любой (обычно JSON/Binary) | Protobuf (Binary) |
| **Поддержка браузеров** | Максимальная (fallback) | Высокая (современные) | Ограниченная (через gRPC-web) |
| **Назначение** | Real-time UI, чаты, уведомления | Низкоуровневые стримы | Межсервисное взаимодействие (microservices) |
