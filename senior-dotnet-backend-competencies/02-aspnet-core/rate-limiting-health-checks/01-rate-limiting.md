# Rate Limiting в ASP.NET Core

Rate Limiting — это стратегия ограничения количества запросов к API, чтобы предотвратить злоупотребление ресурсами, защитить сервер от DoS-атак и обеспечить стабильную работу для всех пользователей.

В .NET 7 появилась встроенная поддержка Rate Limiting через Middleware.

## Основные алгоритмы

### 1. Fixed Window (Фиксированное окно)
Запросы считаются за фиксированный интервал времени. Если лимит исчерпан, все последующие запросы до начала нового окна отклоняются.
- **Пример:** 100 запросов в минуту. В 12:00:00 началось окно, в 12:00:59 сделан 100-й запрос. 101-й запрос в 12:00:59 будет отклонен. В 12:01:00 счетчик обнулится.
- **Проблема:** «Всплеск» трафика на границе окон (двойной лимит за короткий промежуток времени).

### 2. Sliding Window (Скользящее окно)
Модификация фиксированного окна. Интервал делится на сегменты. При проверке лимита учитываются запросы из текущего и предыдущего сегментов. Это сглаживает всплески трафика.
- **Пример:** Если окно 1 минута, оно может состоять из 6 сегментов по 10 секунд.

### 3. Token Bucket (Корзина токенов)
В корзину с постоянной скоростью добавляются «токены». Каждый запрос «тратит» один токен. Если корзина пуста — запрос отклоняется.
- **Особенность:** Позволяет обрабатывать кратковременные всплески (bursts), если в корзине накопились токены.

### 4. Concurrency (Конкурентность)
Ограничивает количество **одновременно** активных запросов. Это не про время, а про объем памяти/потоков.
- **Пример:** Не более 10 активных соединений для тяжелых отчетов.

## Реализация в .NET 7+

Регистрация происходит в `Program.cs`:

```csharp
builder.Services.AddRateLimiter(options => {
    options.AddFixedWindowLimiter(policyName: "fixed", options => {
        options.PermitLimit = 10;
        options.Window = TimeSpan.FromSeconds(10);
        options.QueueLimit = 2; // Сколько запросов подождать в очереди перед 429
        options.QueueRequestLimits = QueueRequestLimits.Unbounded;
    });
});

// Использование
app.UseRateLimiter();

app.MapGet("/api/data", () => "Hello").RequireRateLimiter("fixed");
```

## RateLimitPartition (Сегментация)

Обычно нам нужно ограничивать каждого пользователя индивидуально (по IP или API-key), а не весь сервер целиком. Для этого используется `PartitionedRateLimiter`.

```csharp
options.AddPolicy("user-policy", httpContext => 
    RateLimitPartition.GetFixedWindowLimiter(
        partitionKey: httpContext.User.Identity?.Name ?? "anonymous",
        factory: _ => new FixedWindowRateLimiterOptions { 
            PermitLimit = 100, 
            Window = TimeSpan.FromMinutes(1) 
        }
    )
);
```

## Рекомендации и Pitfalls

- **Код ответа:** По умолчанию сервер возвращает `429 Too Many Requests`. Настройте `options.OnRejected` для кастомного ответа.
- **Распределенный Rate Limiting:** Встроенный механизм работает **в памяти одного инстанса**. Если у вас 5 серверов за балансировщиком, пользователь сможет сделать $5 \times \text{лимит}$ запросов. Для глобального ограничения используйте Redis (библиотеки типа `AspNetCoreRateLimit`).
- **Очереди (QueueLimit):** Будьте осторожны с очередями. Слишком большая очередь может привести к накоплению запросов и резкому увеличению задержки (latency).
