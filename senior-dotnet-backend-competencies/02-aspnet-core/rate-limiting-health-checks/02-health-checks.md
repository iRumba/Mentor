# Health Checks в ASP.NET Core

Health Checks (проверки здоровья) — это механизм, позволяющий внешним системам (Kubernetes, Consul, балансировщикам нагрузки) понять, жив ли экземпляр приложения и готово ли оно обрабатывать запросы.

## Типы проверок (Health Check Types)

В современной инфраструктуре (особенно в K8s) выделяют два основных типа проб:

### 1. Liveness Probe (Жив ли сервер?)
Проверяет, не завис ли процесс приложения. Если Liveness-проверка возвращает ошибку, оркестратор **перезагружает** контейнер.
- **Что проверять:** Базовые вещи. Если `/health/live` отвечает — процесс жив.
- **Важно:** Не добавляйте сюда проверку БД. Если БД временно недоступна, перезагрузка 100 контейнеров не решит проблему, а только создаст «шторм» при старте.

### 2. Readiness Probe (Готов ли сервер?)
Проверяет, может ли приложение выполнять свою работу. Если Readiness-проверка возвращает ошибку, оркестратор **перестает направлять трафик** на этот экземпляр, но не перезагружает его.
- **Что проверять:** Соединение с БД, доступность кэша (Redis), связь с критически важными внешними API.
- **Сценарий:** Приложение запустилось, но БД еще прогревается или мигрирует. Readiness вернет ошибку $\rightarrow$ трафик не пойдет $\rightarrow$ пользователь не увидит 500 ошибку.

## Реализация в .NET

Регистрация в `Program.cs`:

```csharp
builder.Services.AddHealthChecks()
    .AddCheck("MainDB", () => CheckDbConnection()) // Кастомная проверка
    .AddSqlServer(connectionString); // Готовая проверка из NuGet

app.MapHealthChecks("/health");
```

### Продвинутая настройка (разные эндпоинты)

```csharp
builder.Services.AddHealthChecks()
    .AddHealthCheck("/health/live", 
       healthCheckPublisher: null, 
       tags: new[] { "live" });

builder.Services.AddHealthChecks()
    .AddHealthCheck("/health/ready", 
       tags: new[] { "ready" });

// Разделение эндпоинтов по тегам
app.MapHealthChecks("/health/live", new HealthCheckOptions {
    Predicate = check => check.Tags.Contains("live")
});
app.MapHealthChecks("/health/ready", new HealthCheckOptions {
    Predicate = check => check.Tags.Contains("ready")
});
```

## Best Practices

- **Кэширование результатов:** Если проверка БД тяжелая, используйте кэширование результата на несколько секунд.
- **Безопасность:** Эндпоинты здоровья могут раскрывать внутреннюю топологию сети или состояние зависимостей. Ограничьте доступ к ним (например, только для внутреннего IP балансировщика).
- **Деградация:** Реализуйте «мягкие» проверки. Если второстепенный сервис (например, сервис статистики) недоступен, Readiness может остаться `Healthy`, но с предупреждением `Degraded`.
