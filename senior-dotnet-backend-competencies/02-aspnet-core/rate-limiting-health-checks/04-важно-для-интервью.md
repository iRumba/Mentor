# Важно для интервью по Rate Limiting, Health Checks и Middleware

Эти темы проверяют понимание того, как приложение взаимодействует с внешней инфраструктурой (K8s, Load Balancers) и как эффективно управлять потоком запросов.

## Ключевые вопросы и ответы

### 1. В чем разница между Rate Limiting и Throttling?
**Ответ:** Часто используются как синонимы, но технически:
- **Rate Limiting** — это жесткое ограничение количества запросов за время (например, «не более 10 req/sec»).
- **Throttling** — это более гибкое управление пропускной способностью. Оно может включать в себя постепенное замедление ответов или приоритезацию некоторых пользователей над другими при высокой нагрузке.

### 2. Live vs Readiness probes в Kubernetes
**Вопрос:** Почему нельзя просто сделать один общий `/health`?
**Ответ:** Потому что у них разные цели. 
- Если `Liveness` упал $\rightarrow$ K8s **убивает** контейнер. Если при этом БД была временно недоступна, K8s убьет все контейнеры приложения, что приведет к полному отказу системы (Cascading Failure).
- Если `Readiness` упал $\rightarrow$ K8s просто **перестает присылать трафик**. Приложение остается живым, и как только БД восстановится, Readiness снова станет `Healthy`, и трафик вернется.

### 3. Как работает InvokeAsync в кастомном Middleware?
**Вопрос:** Как передать зависимости в Middleware?
**Ответ:** Класс Middleware создается как Singleton. Поэтому зависимости, которые живут меньше (Scoped), нельзя внедрять в конструктор. Их нужно внедрять непосредственно в метод `InvokeAsync(HttpContext context, IMyScopedService service)`. ASP.NET Core автоматически разрешит их из DI для каждого запроса.

### 4. Что такое терминальный Middleware?
**Ответ:** Это middleware, который не вызывает `next()`. Он завершает обработку запроса и возвращает ответ клиенту. Пример — `app.Run()`.

### 5. Как реализовать распределенный Rate Limiting?
**Ответ:** Встроенный в .NET Rate Limiter работает в памяти одного сервера. Для распределенной системы нужно использовать внешнее хранилище счетчиков, например Redis. Можно использовать сторонние библиотеки (например, `AspNetCoreRateLimit`) или API-Gateway (Ocelot, Azure API Management, Nginx), которые делают Rate Limiting на уровне инфраструктуры до того, как запрос попадет в приложение.

## Сценарий-задача: «Дизайн системы защиты API»
**Задача:** «Нам нужно защитить публичный API от злоупотреблений, при этом VIP-пользователям нужно дать в 10 раз больше лимитов».
**Правильный ответ:**
1. Использовать `PartitionedRateLimiter`.
2. В ключе партиции (`partitionKey`) использовать ID пользователя.
3. В фабрике лимитера проверять роль пользователя в `HttpContext.User`.
4. Если роль `VIP` $\rightarrow$ установить `PermitLimit = 1000`, иначе `100`.
5. Установить `QueueLimit = 0` для обычных пользователей (сразу 429) и небольшую очередь для VIP, чтобы сгладить всплески.
