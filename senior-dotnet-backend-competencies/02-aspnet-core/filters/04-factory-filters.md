## Factory Filters

В ASP.NET Core стандартные фильтры-атрибуты имеют ограничение: они создаются как метаданные и не поддерживают внедрение зависимостей (Dependency Injection) в конструктор напрямую, если они используются как `[MyFilter]`.

### Проблема классического атрибута
Если вы создадите класс `MyFilterAttribute : ActionFilterAttribute`, вы не сможете добавить в его конструктор `IDbContext`, так как атрибут создается компилятором в момент компиляции, а не контейнером DI в рантайме.

### Решение: `IFilterFactory` (или `ServiceFilter` / `TypeFilter`)

Для решения этой проблемы существуют три основных механизма:

#### 1. `ServiceFilterAttribute`
Позволяет использовать фильтр, который зарегистрирован в DI-контейнере.
- **Регистрация**: Нужно добавить фильтр в `IServiceCollection`.
- **Использование**: `[ServiceFilter(typeof(MyActionFilter))]`.
- **Плюс**: Полный доступ к DI.
- **Минус**: Нужно не забывать регистрировать фильтр в `Program.cs`.

#### 2. `TypeFilterAttribute`
Похож на `ServiceFilter`, но не требует регистрации фильтра в DI. Он использует `ObjectFactory` для создания экземпляра.
- **Использование**: `[TypeFilter(typeof(MyActionFilter))]`.
- **Плюс**: Не загромождает `Program.cs`.

#### 3. `IFilterFactory` (Custom Filter Factory)
Самый гибкий подход. Вы создаете фабрику, которая сама решает, какой фильтр создать.

```csharp
public class MyFilterFactory : IFilterFactory
{
    public IFilter CreateFilter()
    {
        // Здесь можно реализовать любую логику создания
        return new MyActionFilter();
    }
}
```

### Сравнительная таблица

| Метод | Регистрация в DI | Доступ к DI | Синтаксис | Когда использовать |
| :--- | :---: | :---: | :--- | :--- |
| **Attribute** | Нет | Нет | `[MyFilter]` | Простая логика без зависимостей |
| **ServiceFilter** | **Обязательно** | Да | `[ServiceFilter(typeof(T))]` | Фильтр с тяжелыми зависимостями |
| **TypeFilter** | Нет | Да | `[TypeFilter(typeof(T))]` | Фильтр с DI, но без регистрации |

### Best Practice для Senior
Рекомендуется использовать `TypeFilter`, если фильтр должен быть переиспользуемым и иметь зависимости, но не должен быть синглтоном внутри всего приложения. Если фильтр является частью глобальной инфраструктуры — `ServiceFilter` + глобальная регистрация.
