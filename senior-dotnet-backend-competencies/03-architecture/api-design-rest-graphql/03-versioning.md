## Версионирование API

### Теоретический базис и Senior-перспектива
Версионирование — это стратегия управления эволюцией API без нарушения обратной совместимости (Breaking Changes). Для Senior-разработчика версионирование — это не только выбор способа передачи версии, но и жизненный цикл API: от `Beta` до `Deprecated` и `Sunset`.

Основной принцип: **Никогда не вноси изменения, которые сломают текущих клиентов**. Любое изменение структуры ответа, удаление полей или изменение обязательности параметров требует новой версии или стратегии плавного перехода.

### Стратегии реализации в .NET
В ASP.NET Core версионирование обычно реализуется с помощью пакета `Asp.Versioning.Http`:

1. **URL-based Versioning** (`/api/v1/users`):
   - Самый явный и популярный способ.
   - Легко кэшировать, легко тестировать.
   - Нарушает идею REST (ресурс должен иметь один URI), но на практике доминирует.

2. **Header-based Versioning** (`X-Api-Version: 1.0` или `Accept: application/vnd.myapi.v1+json`):
   - Соответствует принципам REST.
   - Позволяет менять версию, не меняя URL.
   - Сложнее в отладке (нужен инструмент для установки заголовков).

3. **Query Parameter Versioning** (`/api/users?api-version=1.0`):
   - Простой способ, часто используемый как fallback вариант.
   - Загрязняет URL.

### Критические компромиссы
- **Дублирование кода**: Поддержка нескольких версий часто ведет к дублированию логики. Решается через использование адаптеров или мапперов на уровне API.
- **Сложность тестирования**: Необходимо поддерживать наборы тестов для каждой активной версии API.
- **Когнитивная нагрузка**: Новые разработчики должны понимать, какая версия является актуальной и какие из них поддерживаются.

### Практические рекомендации
**Когда использовать:**
- Когда API используется внешними клиентами, которых вы не контролируете.
- При необходимости внедрения радикальных изменений в структуру данных.

**Когда избегать (или упростить):**
- Во внутреннем взаимодействии микросервисов (лучше использовать контрк tract-first подход с gRPC или просто согласованный деплой).
- В MVP-проектах, где API меняется ежедневно (лучше использовать одну версию и договариваться о breaking changes).

**Золотое правило Senior-разработчика:**
Предпочитайте **аддитивные изменения** (добавление новых полей) вместо создания новых версий. Если поле больше не нужно, пометьте его как `[Obsolete]` (в коде) и задокументируйте как `Deprecated` в API, но не удаляйте его, пока все клиенты не перейдут на новую версию.
