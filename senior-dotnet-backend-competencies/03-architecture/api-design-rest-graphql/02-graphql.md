## GraphQL

### Теоретический базис и Senior-перспектива
GraphQL — это язык запросов и среда выполнения (runtime) для API. В отличие от REST, он переносит управление структурой данных от сервера к клиенту. 

С позиций Senior-архитектора, GraphQL решает проблему "недостаточных или избыточных данных" (Underfetching/Overfetching), позволяя клиенту за один запрос получить точно те данные, которые нужны для текущего экрана или функционала. Это критически важно для мобильных приложений и сложных фронтендов с множеством взаимосвязанных сущностей.

### Реализация в .NET
Основным инструментом в экосистеме .NET является библиотека **HotChocolate**:
- **Schema-First vs Code-First**: Возможность описать схему в `.graphql` файлах или автоматически генерировать её из C#-классов.
- **Resolvers**: Функции, которые определяют, как получать конкретное поле.
- **DataLoader**: Паттерн для решения проблемы N+1. Вместо выполнения отдельного запроса в БД для каждой связанной сущности, DataLoader собирает ID и выполняет один пакетный запрос (batching).
- **Integration with EF Core**: Эффективное использование проекций (`Select`) для перевода GraphQL-запроса в оптимальный SQL.

### Критические компромиссы
- **Сложность реализации**: Порог входа выше, чем у REST. Нужно управлять схемой, типами и резолверами.
- **Кэширование**: Стандартное HTTP-кэширование по URL не работает, так как все запросы идут на `/graphql`. Кэширование переносится на уровень клиента (Apollo Client, Relay).
- **Безопасность и DoS**: Клиент может отправить "злонамеренный" глубоко вложенный запрос, который положит сервер. Требуется внедрение анализа сложности запросов (Query Complexity Analysis) и ограничение глубины вложенности.

### Практические рекомендации
**Когда использовать:**
- Сложные интерфейсы с глубокой вложенностью данных.
- При наличии множества различных клиентов (web, ios, android) с разными требованиями к данным.
- API-шлюзы (BFF — Backend for Frontend), агрегирующие данные из нескольких микросервисов.

**Когда избегать:**
- Простые CRUD приложения.
- Системы, где критически важно стандартное HTTP-кэширование на уровне прокси/CDN.
- Команды, не готовые к дополнительному оверхеду в поддержке схемы.

### Сравнение с REST
| Характеристика | REST | GraphQL |
| :--- | :--- | :--- |
| **Кол-во запросов** | Часто несколько (roundtrips) | Один (single request) |
| **Структура данных** | Определяет сервер | Определяет клиент |
| **Типизация** | Слабая/Документированная | Строгая (Schema) |
| **Ошибки** | HTTP статусы (400, 404, 500) | HTTP 200 с массивом `errors` в теле |
