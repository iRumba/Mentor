## REST (Representational State Transfer)

### Теоретический базис и Senior-перспектива
REST — это архитектурный стиль, а не протокол. Его суть в работе с **ресурсами**, идентифицируемыми через URI. Для Senior-разработчика REST — это не просто набор HTTP-методов, а контракт, обеспечивающий единообразие (Uniform Interface), отсутствие состояния на сервере (Stateless) и возможность кэширования.

Главная цель REST — создание предсказуемого интерфейса, где клиент может взаимодействовать с системой, зная общие правила, без глубокого изучения каждого конкретного эндпоинта.

### Реализация в .NET
Современный стек ASP.NET Core предоставляет мощные средства для реализации REST:
- **Minimal APIs / Controllers**: Использование `[HttpGet]`, `[HttpPost]` и др. для маппинга методов.
- **Problem Details (RFC 7807)**: Стандартизация ответов об ошибках через `ProblemDetails`.
- **Routing**: Использование атрибутивного роутинга для создания иерархических ресурсов (например, `/orders/{orderId}/items/{itemId}`).
- **Content Negotiation**: Автоматический выбор формата ответа (JSON, XML) через `Accept` заголовок.

### Критические компромиссы
- **Сложность vs. Простота**: Строгое соблюдение всех принципов (особенно HATEOAS) часто избыточно и усложняет как сервер, так и клиент.
- **Overfetching / Underfetching**: Главная проблема REST — передача лишних данных или необходимость делать несколько запросов для получения связанных сущностей.
- **Maintainability**: Четкое разделение на ресурсы облегчает поддержку, но при разрастании API управление версиями становится критическим вызовом.

### Практические рекомендации
**Когда использовать:**
- Публичные API, которые должны быть максимально доступны и предсказуемы.
- Системы с интенсивным использованием HTTP-кэширования (CDN, браузеры).
- Простые CRUD-сервисы с понятной иерархией ресурсов.

**Когда избегать:**
- При необходимости получения данных из множества разных источников одним запросом (здесь лучше GraphQL).
- В высоконагруженных внутренних микросервисах, где overhead HTTP/JSON слишком велик (лучше gRPC).
- При реализации real-time обновлений (лучше WebSocket/SignalR).

### Сравнение с другими стилями
| Критерий | REST | GraphQL | gRPC |
| :--- | :--- | :--- | :--- |
| **Передача данных** | Фиксированные ответы | Клиент определяет набор полей | Строго типизированный бинарный протокол |
| **Кэширование** | На уровне HTTP (стандарт) | Сложное (на уровне клиента/полей) | Не поддерживается стандартно |
| **Типизация** | Опционально (OpenAPI/Swagger) | Строгая встроенная схема | Строгая (Protocol Buffers) |
| **Связанность** | Слабая (через URI) | Сильная (граф зависимостей) | Сильная (контракт `.proto`) |
