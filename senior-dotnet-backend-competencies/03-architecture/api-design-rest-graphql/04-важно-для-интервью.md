## Важные аспекты API Design для интервью

Этот раздел объединяет ключевые архитектурные вопросы, которые часто задают на позицию Senior .NET Backend разработчика.

### REST vs GraphQL vs gRPC: Когда и что?
На интервью важно показать, что вы не фанат одного инструмента, а прагматичный архитектор.
- **REST** — стандарт для публичных веб-интерфейсов. Выбирайте его, когда нужны предсказуемость, простота и стандартное HTTP-кэширование.
- **GraphQL** — идеален для сложных фронтендов (BFF), когда клиент сам должен определять состав данных. Выбирайте его для снижения количества запросов и борьбы с overfetching.
- **gRPC** — лучший выбор для внутреннего взаимодействия микросервисов. Выбирайте его для максимальной производительности, строгой типизации и использования HTTP/2.

### a Breaking Change — что это и как с этим жить?
**Определение:** Любое изменение в API, которое приводит к ошибке у клиента, который не обновлял свой код.
- **Примеры:** Удаление поля, изменение типа данных поля, добавление нового обязательного параметра в запрос, изменение формата даты.
- **Как избежать:**
    1. Только аддитивные изменения.
    2. Постепенный вывод из эксплуатации (Deprecation Policy).
    3. Строгое версионирование.

### Идемпотентность API
Критически важная тема для Senior.
- **Что это:** Способность метода при повторном вызове с теми же параметрами давать тот же результат и не изменять состояние системы более одного раза.
- **Методы по умолчанию:** GET, PUT, DELETE — идемпотентны. POST — нет.
- **Как реализовать идемпотентность для POST:** Использование `Idempotency-Key` в загологах. Сервер запоминает ключ и результат операции. Если приходит запрос с тем же ключом, сервер просто возвращает сохраненный результат без повторного выполнения бизнес-логики.

### Кэширование на разных уровнях
- **Client-side**: `Cache-Control` заголовки.
- **Edge (CDN/Proxy)**: Промежуточное кэширование ответов.
- **Server-side**: `IMemoryCache`, `IDistributedCache` (Redis), кэширование на уровне БД.
- **Инвалидация**: Самая сложная часть. Стратегии: TTL (Time-to-live) или событийная инвалидация (через шину событий).

### Безопасность API
- **Аутентификация/Авторизация**: OAuth2, OpenID Connect, JWT (понимание структуры, подписи, срока действия).
- **Rate Limiting**: Ограничение количества запросов для защиты от DoS и расхода ресурсов. В .NET 7+ есть встроенный `RateLimiter` middleware.
- **Валидация**: Использование FluentValidation для проверки входящих данных до того, как они попадут в бизнес-логику.
