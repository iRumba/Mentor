## Clean Architecture (Чистая архитектура)

### Теоретический базис и Senior-перспектива
Чистая архитектура (по Роберту Мартину) развивает идеи луковой и гексагональной архитектур, вводя более строгие правила разделения и акцент на **Use Cases (Сценарии использования)**.

Главная цель — сделать систему максимально независимой от деталей реализации. Правило зависимости гласит: зависимости могут быть направлены только внутрь, к более стабильным высокоуровневым политикам.

### Основные компоненты
1. **Entities (Сущности)**: Бизнес-правила самого высокого уровня (Enterprise Wide Business Rules). Это объекты, которые остаются неизменными независимо от приложения.
2. **Use Cases (Сценарии)**: Специфичные для приложения правила (Application Specific Business Rules). Координируют поток данных из и к сущностям.
3. **Interface Adapters**: Преобразователи данных из формата, удобного для Use Cases, в формат, удобный для внешних агентов (Kontrollers, Presenters, Gateways).
4. **Frameworks & Drivers**: Самый внешний слой. Здесь живут БД, UI-фреймворки, веб-серверы.

### Реализация в .NET
В экосистеме .NET Clean Architecture часто реализуется с использованием паттерна **MediatR**:
- **Commands/Queries**: Описывают Use Case.
- **Handlers**: Содержат логику выполнения конкретного сценария.
- **Entities**: POCO классы в ядре.
- **Persistence**: Реализация репозиториев в отдельном проекте, который зависит от слоя Use Cases.

### Критические компромиссы
- **Избыточность**: Огромный объем кода для простых операций.
- **Сложность внедрения**: Потребуется серьезное переобучение команды, если они привыкли к классическому MVC.

### Практические рекомендации
**Когда использовать:**
- Крупные проекты с большим количеством бизнес-сценариев.
- Когда требуется строгий контроль над тем, как бизнес-логика взаимодействует с внешним миром.

**Когда избегать:**
- В микросервисах с ограниченной ответственностью (Single Responsibility). Если сервис делает только одну вещь, Clean Architecture превратит его в "монстра" из пяти проектов.
