## Onion Architecture (Луковая архитектура)

### Теоретический базис и Senior-перспектива
Луковая архитектура представляет собой систему как набор концентрических кругов. Главный постулат: **Зависимости направлены только внутрь**. Внутренние слои ничего не знают о внешних.

Центром системы является Domain Model. Все остальные слои служат лишь для поддержки этой модели и её реализации.

### Структура слоев (от центра к периферии)
1. **Domain Model (Ядро)**:
   - Содержит сущности (Entities), Value-объекты, исключения домена.
   - Не имеет внешних зависимостей.
2. **Domain Services**:
   - Логика, которая не помещается в одну сущность.
   - Определяет интерфейсы репозиториев.
3. **Application Services**:
   - Реализует конкретные сценарии использования (Use Cases).
   - Оркестрирует работу доменных моделей и сервисов.
4. **Infrastructure (Инфраструктура)**:
   - Реализация репозиториев (EF Core, Dapper).
   - Реализация внешних интеграций (SMS, Email, Cloud Storage).
5. **UI / Presentation (API)**:
   - Точка входа. Контроллеры, SignalR хабы, gRPC сервисы.

### Реализация в .NET
В .NET луковая архитектура часто воплощается в структуре проектов:
- `Project.Domain` $\rightarrow$ `Project.Application` $\rightarrow$ `Project.Infrastructure` $\rightarrow$ `Project.WebAPI`.
- При этом `Infrastructure` и `WebAPI` реализуют интерфейсы, определенные в `Domain` и `Application`, что и обеспечивает инверсию зависимостей через DI-контейнер.

### Критические компромиссы
- **Over-engineering**: Для простых задач такая структура избыточна.
- **Сложность навигации**: Чтобы добавить одно поле в API, нужно изменить модель в Core, DTO в Application, маппинг и контроллер в API.

### Практические рекомендации
**Когда использовать:**
- Когда приложение имеет четкое разделение на бизнес-логику и техническую инфраструктуру.
- В крупных командах, где разные люди работают над разными слоями.

**Когда избегать:**
- В небольших сервисах, где бизнес-логика сводится к простому перемещению данных из БД в JSON.
