## Hexagonal Architecture (Порты и адаптеры)

### Теоретический базис и Senior-перспектива
Гексагональная архитектура фокусируется на взаимодействии приложения с внешним миром. Основная идея в том, что приложение должно быть "изолированным ядром", которое общается с внешним миром через строго определенные **Порты**.

Для Senior-разработчика это означает, что приложение не должно знать, кто его вызывает (человек через браузер, другой сервис через очередь или автотест) и куда оно пишет данные (SQL, NoSQL или текстовый файл).

### Ключевые концепции
- **The Core (Ядро)**: Содержит бизнес-логику.
- **Ports (Порты)**: Интерфейсы, определяющие, *как* с ядром можно взаимодействовать. Бывают двух типов:
    - **Inbound (Driving) Ports**: API, которые ядро предоставляет внешнему миру (например, `IOrderService`).
    - **Outbound (Driven) Ports**: Интерфейсы, за которыми ядро обращается к внешним системам (например, `IOrderRepository`).
- **Adapters (Адаптеры)**: Конкретные реализации портов.
    - **Driving Adapters**: Преобразует внешний запрос в вызов порта (например, `OrderController` вызывает `IOrderService`).
    - **Driven Adapters**: Реализует порт для взаимодействия с внешней системой (например, `SqlOrderRepository` реализует `IOrderRepository`).

### Реализация в .NET
В .NET эта архитектура реализуется через четкое именование и разделение ответственности:
- **Domain/Application** $\rightarrow$ Определяют интерфейсы (Порты).
- **Infrastructure/Web** $\rightarrow$ Создают классы-реализации (Адаптеры).
- Использование `Microsoft.Extensions.DependencyInjection` позволяет легко подменять адаптеры (например, заменить `MockRepository` на `SqlRepository` в конфигурации).

### Критические компромиссы
- **Повторение структур**: Часто приходится создавать почти идентичные модели данных для разных типов адаптеров.
- **Сложность отладки**: Из-за высокого уровня абстракции путь запроса от контроллера до БД проходит через множество слоев и интерфейсов.

### Практические рекомендации
**Когда использовать:**
- Когда приложение должно поддерживать несколько способов взаимодействия (например, и REST API, и CLI-интерфейс, и обработчик сообщений из Kafka).
- Когда инфраструктура (БД, сторонние API) часто меняется или находится в стадии неопределенности.

**Когда избегать:**
- Когда есть только один способ взаимодействия (один API $\rightarrow$ одна БД). В этом случае гексагональная архитектура превращается в обычный трехслойный подход с лишними интерфейсами.
