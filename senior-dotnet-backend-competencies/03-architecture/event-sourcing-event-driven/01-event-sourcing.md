## Event Sourcing (Событийно-ориентированное хранение)

### Теоретический базис и Senior-перспектива
Event Sourcing — это парадигма хранения данных, при которой состояние объекта не перезаписывается, а представляется как последовательность неизменяемых событий. 

С точки зрения архитектора, это переход от хранения "Снапшота" (текущего состояния) к хранению "Журнала изменений" (Log). Основная ценность здесь — **полная прослеживаемость (Traceability)**. Мы не просто знаем, что пользователь сменил адрес, мы знаем, когда, почему, и какой адрес был до этого.

### Ключевые компоненты
1. **Event Store (Хранилище событий)**:
   - Специализированная БД (например, EventStoreDB, Marten) или таблица в SQL.
   - Работает только в режиме `Append-Only`. События никогда не редактируются и не удаляются.
2. **Events (События)**:
   - Представляют факт, который уже произошел (`OrderPaid`, `UserRegistered`).
   - Должны быть иммутабельными (Immutable).
3. **Projections (Проекции)**:
   - Процесс превращения потока событий в конкретную модель данных (Read Model).
   - Пример: Проектор слушает все события `MoneyDeposited` и `MoneyWithdrawn` и обновляет текущий баланс пользователя в SQL-таблице.
4. **Snapshots (Снапшоты)**:
   - Чтобы не перечитывать миллион событий при каждом запросе, система периодически сохраняет текущее состояние объекта.

### Реализация в .NET
- **Marten**: Позволяет использовать PostgreSQL как полноценный Event Store с поддержкой документов и событий.
- **EventStoreDB**: Специализированная БД для событий.
- **Своя реализация**: Создание таблицы `Events` с колонками `AggregateId`, `Version`, `EventType`, `Payload (JSON)`.

### Критические компромиссы
- **Сложность чтения**: Чтобы узнать текущее состояние, нужно "проиграть" (replay) все события. Это решается снапшотами.
- **Сложность обновления данных**: Если в структуре события была ошибка, нельзя просто сделать `UPDATE`. Нужно создавать новое событие-коррекцию или использовать Upcasting.
- **Eventual Consistency**: Данные в проекциях обновляются с задержкой.

### Практические рекомендации
**Когда использовать:**
- В финансовых системах (банковские выписки), где аудит обязателен по закону.
- В системах с очень сложными правилами изменения состояния.
- Когда нужно иметь возможность "отмотать" состояние системы на любую дату.

**Когда избегать:**
- В стандартных CRUD-приложениях. ES добавляет огромный слой сложности, который не приносит пользы в простых сценариях.
