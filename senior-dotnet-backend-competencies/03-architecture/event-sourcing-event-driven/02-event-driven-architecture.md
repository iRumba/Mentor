## Event-Driven Architecture (EDA)

### Теоретический базис и Senior-перспектива
Event-Driven Architecture (Событийно-ориентированная архитектура) — это стиль проектирования, где потоки событий являются основным средством коммуникации между сервисами.

В отличие от REST (запрос-ответ), EDA переходит к модели **асинхронного уведомления**. Сервис не говорит: "Сделай это", он говорит: "Произошло вот это, делайте с этим что хотите". Это позволяет достичь максимальной слабой связанности (Loose Coupling).

### Основные концепции
1. **Event Bus / Message Broker**:
   - Посредник, который принимает события от издателей и доставляет их подписчикам.
   - Примеры: **RabbitMQ** (умный брокер, глупый потребитель), **Apache Kafka** (глупый брокер, умный потребитель/лог событий), **Azure Service Bus**.
2. **Patterns of Communication**:
   - **Publish/Subscribe (Pub/Sub)**: Издатель отправляет событие в "топик", и все заинтересованные подписчики его получают.
   - **Point-to-Point**: Сообщение предназначено конкретному получателю (очередь).
3. **Типы событий**:
   - **Domain Events**: Важны внутри одного контекста (например, `OrderValidated`).
   - **Integration Events**: Предназначены для синхронизации других микросервисов (например, `OrderPlaced` $\rightarrow$ вызывает сервис оплаты и сервис склада).

### Реализация в .NET
Лучшим инструментом для реализации EDA в .NET является **MassTransit**:
- **Абстракция над брокерами**: Позволяет переключаться между RabbitMQ, Azure Service Bus и Amazon SQS без изменения бизнес-логики.
- **Sagas**: Управление сложными распределенными транзакциями (State Machine).
- **Outbox Pattern**: Гарантирует, что событие будет отправлено в брокер только если данные были успешно сохранены в БД.

### Критические компромиссы
- **Сложность отладки**: Трассировка запроса через 5 разных сервисов и 3 очереди — настоящий кошмар без распределенного логирования (OpenTelemetry).
- **Гарантии доставки**: Нужно решать, что делать с сообщениями, которые пришли дважды (At-least-once delivery) или пропали.
- **Отсутствие мгновенного ответа**: Пользователь не получает результат сразу, что требует изменения UX (например, уведомления через WebSocket или SignalR).

### Практические рекомендации
**Когда использовать:**
- В распределенных микросервисах для снижения связанности.
- При реализации фоновых процессов, которые не должны блокировать пользователя (например, генерация тяжелого PDF-отчета).
- Для интеграции с внешними системами, которые работают асинхронно.

**Когда избегать:**
- Внутри одного монолита, если можно обойтись простыми событиями в памяти (C# events).
- В системах, где требуется мгновенная синхронная согласованность (Strong Consistency) на всех этапах.
