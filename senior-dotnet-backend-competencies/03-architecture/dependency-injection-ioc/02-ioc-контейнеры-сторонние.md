## Сторонние IoC-контейнеры и расширения

### Зачем сторонние контейнеры, если есть встроенный Microsoft.Extensions.DependencyInjection?
Встроенный DI в .NET максимально прост и эффективен, но в крупных Enterprise-проектах его функционала часто становится недостаточно. Senior-архитектор выбирает сторонние инструменты, когда требуются сложные сценарии регистрации.

### Популярные решения и их ценность
1. **Autofac**:
   - **Декораторы**: Позволяет обернуть сервис в другой класс без изменения кода самого сервиса (например, добавить логирование).
   - **Property Injection**: Мощная поддержка внедрения через свойства.
   - **Modules**: Позволяют группировать регистрации по функциональным блокам, что упрощает поддержку в огромных проектах.
   - **Interception**: Интеграция с Castle DynamicProxy для реализации AOP (Aspect Oriented Programming) прямо из контейнера.

2. **Scrutor**:
   - Это не отдельный контейнер, а библиотека-расширение для встроенного DI.
   - **Assembly Scanning**: Позволяет автоматически регистрировать все классы, соответствующие определенному паттерну (например, все классы, заканчивающиеся на `Repository`), избавляя от сотен строк ручных регистраций в `Program.cs`.
   - **Decorators**: Упрощает реализацию паттерна "Декоратор" над стандартным DI.

### Критические компромиссы
- **Оверхед по памяти/CPU**: Сторонние контейнеры (особенно с AOP/Interception) могут быть чуть медленнее встроенного.
- **Сложность отладки**: Ошибки при разрешении зависимостей в сложных графах Autofac могут быть труднее анализировать, чем в стандартном DI.

### Практические рекомендации
**Когда использовать встроенный DI**:
- В большинстве микросервисов.
- Когда зависимости просты и прозрачны.

**Когда переходить на Autofac/Scrutor**:
- В больших монолитах с сотнями сервисов.
- При необходимости внедрения сквозной функциональности (AOP) без изменения бизнес-кода.
- Когда ручная регистрация зависимостей становится узким местом в разработке.
