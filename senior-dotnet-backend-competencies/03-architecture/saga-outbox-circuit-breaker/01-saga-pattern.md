# Saga Pattern: Распределенные транзакции в микросервисах

## Теоретический фундамент
В архитектуре микросервисов невозможно использовать классические ACID-транзакции (2PC — Two-Phase Commit из-за проблем с производительностью и доступностью). Паттерн Saga решает задачу обеспечения **согласованности в конечном счете (Eventually Consistency)**.

Saga представляет собой последовательность локальных транзакций. Если одна из транзакций завершается ошибкой, Saga запускает серию **компенсирующих транзакций**, чтобы откатить изменения, внесенные предыдущими шагами.

---

## Виды реализации Saga

### 1. Choreography (Хореография)
В этом подходе нет центрального управляющего. Каждый сервис слушает события от других сервисов и решает, нужно ли ему выполнить действие.

- **Механизм:** 
    1. Сервис А выполняет локальную транзакцию $\rightarrow$ публикует событие `OrderCreated`.
    2. Сервис Б слушает `OrderCreated` $\rightarrow$ резервирует товар $\rightarrow$ публикует `ProductReserved`.
    3. Сервис В слушает `ProductReserved` $\rightarrow$ списывает деньги $\rightarrow$ публикует `PaymentProcessed`.
- **Плюсы:** Высокая степень развязки (Decoupling), нет единой точки отказа.
- **Минусы:** Сложно отслеживать общий ход процесса (Distributed Trace), риск циклической зависимости событий.

### 2. Orchestration (Оркестрация)
Существует центральный объект — Оркестратор (Saga Manager), который говорит участникам, что делать и в какой последовательности.

- **Механизм:** 
    1. Оркестратор получает запрос $\rightarrow$ посылает команду «Зарезервируй товар» в Сервис Б.
    2. Сервис Б отвечает «Успешно».
    3. Оркестратор посылает команду «Спиши деньги» в Сервис В.
- **Плюсы:** Централизованное управление, проще отладка и визуализация процесса, нет циклов.
- **Минусы:** Риск превращения Оркестратора в «Божественный объект» с излишней бизнес-логикой, дополнительная точка отказа.

## Реализация в .NET

Для реализации Saga в .NET наиболее популярными инструментами являются:
- **MassTransit:** Имеет встроенную поддержку State Machines (Saga State Machine), которая позволяет декларативно описать состояния и переходы.
- **NServiceBus:** Мощный инструмент для управления распределенными процессами.
- **Temporal.io:** Более тяжелое решение для долгоживущих рабочих процессов (Workflows).

**Пример логики компенсации:**
Если на шаге «Оплата» произошла ошибка $\rightarrow$ Оркестратор (или Хореография) посылает сигнал в Сервис Б $\rightarrow$ `CancelProductReservation` (возврат товара на склад).

## Trade-offs (Плюсы и Минусы)

| Плюсы | Минусы |
| :--- | :--- |
| **Масштабируемость:** Позволяет выполнять длинные бизнес-процессы без блокировок ресурсов. | **Сложность реализации:** Компенсирующие транзакции должны быть идемпотентными и обрабатывать все крайние случаи. |
| **Отказоустойчивость:** Система продолжает работать, даже если один из сервисов временно недоступен. | **Отсутствие изоляции:** Другие транзакции могут видеть промежуточные (незавершенные) данные. |

## Когда использовать (и когда НЕ использовать)

**Использовать:**
- В распределенных системах, где бизнес-процесс охватывает несколько независимых БД.
- Когда время выполнения транзакции может быть долгим (например, ожидание оплаты от внешнего шлюза).

**Не использовать:**
- В монолитах (используйте обычные транзакции БД).
- Если бизнес-требования требуют строгой изоляции (Strong Consistency) в режиме реального времени.

## Сравнение с альтернативами
Альтернативой является **2PC (Two-Phase Commit)**, но он практически не используется в современном облачном бэкенде из-за блокировок и низкой доступности (Availability).

## Best Practices в .NET
1. **Идемпотентность:** Каждый обработчик события в Saga должен быть идемпотентным (повторный вызов с тем же ID не должен привести к дублированию действия).
2. **Outbox Pattern:** Всегда используйте Outbox вместе с Saga, чтобы гарантировать, что событие будет отправлено после успешного сохранения локальной транзакции.
3. **Мониторинг:** Используйте корреляционные ID (Correlation ID), чтобы проследить путь одной Saga через все микросервисы в Jaeger или Zipkin.
