# Circuit Breaker: Предохранитель для отказоустойчивых систем

## Теоретический фундамент
В распределенных системах один упавший сервис может вызвать «эффект домино» (каскадный сбой). Если Сервис А вызывает Сервис Б, который перегружен или недоступен, все потоки в Сервисе А будут ждать тайм-аута, что приведет к исчерпанию ресурсов (потоков, памяти) самого Сервиса А.

**Circuit Breaker** — это паттерн, который «размыкает цепь», если количество ошибок при вызовах внешнего сервиса превышает определенный порог. Вместо того чтобы пытаться вызвать упавший сервис, приложение мгновенно возвращает ошибку или «запасной» ответ (**Fallback**).

---

## Состояния предохранителя

1. **Closed (Закрыто):** Нормальное состояние. Запросы проходят к сервису. Если количество ошибок достигает порога $\rightarrow$ переход в состояние **Open**.
2. **Open (Открыто):** Все запросы мгновенно отклоняются (Fast Fail). В этом состоянии система «дает отдохнуть» упавшему сервису. Через определенное время (timeout) происходит переход в **Half-Open**.
3. **Half-Open (Полуоткрыто):** Пропускается ограниченное количество пробных запросов. 
    - Если запросы успешны $\rightarrow$ переход обратно в **Closed**.
    - Если есть ошибка $\rightarrow$ переход снова в **Open**.

## Реализация в .NET с помощью Polly

Библиотека **Polly** является стандартом де-факто для реализации устойчивости (resilience) в .NET.

### Основные политики:
- **Retry Policy:** Повторяет запрос несколько раз. Эффективна при «мигающих» ошибках (Network Glitch).
- **Circuit Breaker Policy:** Прекращает вызовы при систематических сбоях.

**Важное различие:** Retry увеличивает нагрузку на и без того упавший сервис, в то время как Circuit Breaker её снимает.

**Пример конфигурации в `Program.cs` (через `AddHttpClient`):**
```csharp
var circuitBreakerPolicy = HttpPolicyExtensions
    .HandleTransientHttpError()
    .CircuitBreakerAsync(
        handledEventsAllowedBeforeBreaking: 5, // Порог ошибок
        durationOfBreak: TimeSpan.FromSeconds(30) // Время восстановления
    );

services.AddHttpClient<IExternalService, ExternalService>()
    .AddPolicyHandler(circuitBreakerPolicy);
```

## Trade-offs (Плюсы и Минусы)

| Плюсы | Минусы |
| :--- | :--- |
| **Предотвращение каскадных сбоев:** Система остается частично работоспособной даже при падении зависимостей. | **Сложность настройки:** Неправильный порог ошибок может вызвать ложные срабатывания. |
| **Быстрый ответ пользователю:** Вместо ожидания 30-секундного тайм-аута пользователь мгновенно получает сообщение об ошибке. | **Необходимость Fallback-логики:** Нужно продумывать, что возвращать, когда «цепь разорвана». |
| **Снижение нагрузки на упавший сервис:** Позволяет сервису восстановиться быстрее. | |

## Когда использовать (и когда НЕ использовать)

**Использовать:**
- При любом синхронном вызове внешнего HTTP/gRPC сервиса.
- В высоконагруженных системах, где доступность (Availability) важнее, чем актуальность данных в конкретный момент.

**Не использовать:**
- В простых приложениях с одной зависимостью, которая гарантированно доступна.
- При выполнении критически важных операций, которые нельзя заменить Fallback-ответом (в этом случае лучше использовать очереди и асинхронность).

## Сравнение с альтернативами
- **Retry:** Помогает при временных сбоях.
- **Bulkhead Isolation:** Ограничивает количество одновременных вызовов к сервису, чтобы один медленный эндпоинт не «съел» все потоки приложения.

## Best Practices в .NET
1. **Используйте Fallback:** Всегда определяйте, что делать при Open state (например, возвращать закэшированные данные, пустой список или сообщение «Сервис временно недоступен»).
2. **Мониторинг состояний:** Подпишитесь на события изменения состояния (`onBreak`, `onReset`), чтобы отправлять алерты в Prometheus/Grafana.
3. **Комбинируйте политики:** Используйте `PolicyWrap` $\rightarrow$ `Fallback` $\rightarrow$ `Circuit Breaker` $\rightarrow$ `Retry`.
