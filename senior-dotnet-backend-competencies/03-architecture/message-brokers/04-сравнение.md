## Сравнение Message Brokers

Выбор брокера зависит не от "популярности", а от конкретных требований к пропускной способности, гарантиям доставки и сложности администрирования.

### Сравнительная таблица

| Характеристика | RabbitMQ | Apache Kafka | Azure Service Bus |
| :--- | :--- | :--- | :--- |
| **Основная модель** | Очереди / Exchanges | Распределенный Лог | Очереди / Топики |
| **Гарантия порядка** | Только 1 consumer на очередь | Внутри партиции | Только в рамках сессий |
| **Хранение данных** | Удаляет после ACK | Retention-based (хранит всё) | TTL-based / Max size |
| **Маршрутизация** | Очень гибкая (Topic, Fanout) | Простая (только по топикам) | Гибкая (SQL-фильтры) |
| **Пропускная способность** | Средняя / Высокая | Экстремально высокая | Высокая |
| **Сложность Ops** | Средняя | Высокая | Низкая (PaaS) |
| **Основной кейс** | Task Queues, RPC | Streaming, Event Sourcing | Enterprise Cloud Apps |

### Резюме для архитектора
- Если вам нужно **"быстро и надежно доставить задачу"** $\rightarrow$ **RabbitMQ**.
- Если вам нужно **"собрать все события мира и уметь их перечитывать"** $\rightarrow$ **Kafka**.
- Если вам нужно **"надежное корпоративное решение в Azure без администрирования"** $\rightarrow$ **Azure Service Bus**.
