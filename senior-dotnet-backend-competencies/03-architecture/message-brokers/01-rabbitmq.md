## RabbitMQ

### Теоретический базис и Senior-перспектива
RabbitMQ — это классический брокер сообщений, реализующий протокол AMQP (Advanced Message Queuing Protocol). Его главная особенность — гибкая маршрутизация сообщений с помощью **Exchanges**.

Для Senior-разработчика RabbitMQ — это инструмент для создания сложных цепочек доставки, где сообщение может быть направлено в одну, несколько или вообще ни в одну очередь в зависимости от настроек маршрутизации.

### Ключевые концепции
- **Producer**: Тот, кто отправляет сообщение.
- **Exchange**: "Почтовый узел", который решает, в какую очередь отправить сообщение.
    - **Direct**: Строгое соответствие ключа маршрутизации (Routing Key).
    - **Topic**: Маршрутизация по маске (например, `orders.*` или `orders.created`).
    - **Fanout**: Пересылка во все привязанные очереди (broadcast).
    - **Headers**: Маршрутизация по атрибутам заголовка.
- **Queue**: Хранилище сообщений, которые ждут обработки потребителем.
- **Consumer**: Тот, кто забирает и обрабатывает сообщение.

### Реализация в .NET
Основной способ работы — библиотека `RabbitMQ.Client` или высокоуровневая обертка **MassTransit**.
- **Acknowledgement (ACK)**: Подтверждение получения сообщения. Позволяет избежать потери данных, если потребитель упал в процессе обработки.
- **Dead Letter Exchange (DLX)**: Очередь для "плохих" сообщений, которые не удалось обработать после N попыток.

### Критические компромиссы
- **Хранение данных**: RabbitMQ предназначен для временного хранения. Если очередь переполняется или сообщения не забираются, это может привести к деградации производительности всего брокера.
- **Порядок сообщений**: Гарантируется только при наличии одного потребителя на одну очередь. При масштабировании (несколько consumer) порядок может быть нарушен.

### Практические рекомендации
**Когда использовать:**
- Для реализации паттерна "Очередь задач" (Task Queues).
- Когда нужна сложная логика маршрутизации.
- Для систем с умеренными нагрузками, где важна надежная доставка каждого сообщения.

**Когда избегать:**
- В системах с экстремальными объемами данных (терабайты в сутки), где лучше подойдет Kafka.
- В сценариях, где нужно "перечитывать" старые сообщения (RabbitMQ удаляет сообщение после ACK).
