## Azure Service Bus (ASB)

### Теоретический базис и Senior-перспективе
Azure Service Bus — это полнофункциональный облачный брокер сообщений, предлагающий баланс между возможностями RabbitMQ и удобством управляемого сервиса (PaaS).

Для Senior-разработчика ASB — это способ быстро развернуть надежный транспорт сообщений, который интегрирован с остальной экосистемой Azure (Active Directory, Monitor, Key Vault).

### Основные возможности
1. **Queues (Очереди)**: Классическая модель "один отправитель $\rightarrow$ один получатель".
2. **Topics & Subscriptions (Топики и Подписки)**: Модель Pub/Sub. Особенность ASB в том, что подписки могут иметь **SQL-фильтры**. Это позволяет подписчику получать только те сообщения, которые соответствуют определенному правилу, прямо на уровне брокера.
3. **Sessions (Сессии)**: Позволяют гарантировать порядок обработки сообщений для конкретной группы (например, все события по одному заказу должны обрабатываться одним и тем же потребителем последовательно).
4. **Scheduled Messages**: Возможность отправить сообщение, которое станет доступным для обработки только через определенное время.
5. **Dead-Lettering**: Автоматический перенос сообщений в специальную очередь при ошибках.

### Реализация в .NET
Используется пакет `Azure.Messaging.ServiceBus`.
- **Integration**: Отлично работает с Azure Functions (триггеры по очереди).
- **Reliability**: Поддержка транзакций (в рамках одного неймспейса) и гарантированная доставка.

### Критические компромиссы
- **Vendor Lock-in**: Ваша архитектура становится зависимой от Azure. Переезд на другой облачный провайдер или on-premise потребует переписывания слоя транспорта.
- **Стоимость**: При очень больших объемах данных облачный брокер может выйти дороже, чем собственный кластер Kafka или RabbitMQ.

### Практические рекомендации
**Когда использовать:**
- В проектах, развернутых в Azure.
- Когда нужна высокая надежность и поддержка сложных функций (сессии, фильтрация) без необходимости администрировать серверы.
- Для интеграции корпоративных систем через стандартные очереди.

**Когда избегать:**
- В проектах, где требуется абсолютная независимость от облачного провайдера.
- При необходимости обработки потоков данных в реальном времени с миллионами событий в секунду (лучше Kafka).
