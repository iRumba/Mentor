## Apache Kafka

### Теоретический базис и Senior-перспектива
Kafka — это не просто брокер сообщений, а **распределенный лог коммитов**. В отличие от RabbitMQ, Kafka не удаляет сообщение сразу после того, как оно было прочитано. Она хранит поток данных в течение заданного времени (retention period).

Для Senior-архитектора Kafka — это фундамент для построения систем реального времени (Streaming), Event Sourcing и высоконагруженных конвейеров данных.

### Ключевые концепции
- **Topic**: Логический поток данных.
- **Partition (Партиция)**: Топик делится на части. Каждая партиция — это упорядоченный лог. Это позволяет масштабировать чтение и запись горизонтально.
- **Offset**: Указатель на последнее прочитанное сообщение. Потребитель сам управляет своим смещением, что позволяет ему "отмотать" данные назад.
- **Consumer Group**: Группа потребителей, которые совместно обрабатывают партиции. Каждая партиция читается только одним потребителем внутри группы, что гарантирует порядок внутри партиции.
- **Zookeeper / KRaft**: Механизмы управления метаданными кластера.

### Реализация в .NET
Основной инструмент — библиотека `Confluent.Kafka`.
- **Producer**: Отправляет данные в топик, выбирая партицию на основе ключа сообщения (Keyed messages).
- **Consumer**: Подписывается на топик и читает сообщения последовательно.
- **Kafka Connect & Kafka Streams**: Мощные инструменты для интеграции с БД и обработки потоков (хотя в .NET чаще используют внешние сервисы или кастомные воркеры).

### Критические компромиссы
- **Сложность настройки**: Kafka требует серьезного администрирования (настройка репликации, партицирования, управления ресурсами).
- **Отсутствие "умной" маршрутизации**: В Kafka нет Exchange. Если вам нужно отправить сообщение в разные места по сложным правилам, вам придется делать это на стороне продюсера или через Kafka Streams.
- **Задержка**: Хотя Kafka очень быстрая, её модель чтения (batching) может давать чуть большую задержку, чем классические очереди.

### Практические рекомендации
**Когда использовать:**
- Для обработки огромных потоков данных (Log aggregation, User tracking).
- Для реализации Event Sourcing.
- Когда нужно, чтобы разные сервисы могли читать одни и те же данные в своем темпе (Replayability).

**Когда избегать:**
- В простых проектах, где достаточно RabbitMQ или даже простой очереди в БД.
- Когда нужна сложная маршрутизация сообщений "на лету".
