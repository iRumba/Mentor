# YAGNI: You Ain't Gonna Need It (Вам это не понадобится)

## Теоретический фундамент
Принцип YAGNI — это философия разработки, которая призывает не внедрять функциональность до тех пор, пока она действительно не станет необходимой. Это борьба с «инстинктом предугадывания», когда разработчик добавляет параметры, интерфейсы или целые модули «на будущее».

---

## Реализация в контексте .NET

### 1. Борьба с преждевременным обобщением
Частая ошибка — создание сложного Generic-механизма или набора абстракций для одного конкретного случая, потому что «завтра может появиться второй тип данных».

- **Пример YAGNI:** Создание сложной системы плагинов для расчета налога, когда в проекте всего одна страна и один вид налога.
- **Правильный подход:** Реализовать простую логику. Если появится вторая страна, провести рефакторинг и вынести общие части в интерфейс.

### 2. Отказ от избыточного кэширования
Разработчики часто внедряют Redis или сложные стратегии кэширования в начале проекта, когда данных еще мало и обычного `IMemoryCache` или даже простого запроса к БД достаточно.

### 3. Ограничение набора параметров в API
Добавление в DTO полей, которые «могут понадобиться фронтенду через месяц», засоряет контракты и усложняет поддержку.

## Trade-offs (Плюсы и Минусы)

| Плюсы | Минусы |
| :--- | :--- |
| **Скорость поставки (Time-to-Market):** Функционал разрабатывается и тестируется быстрее. | **Необходимость рефакторинга:** При появлении новой задачи придется переписывать часть кода. |
| **Чистота кода:** Меньше лишних классов, методов и переменных. | **Риск упустить архитектурную возможность:** Иногда слишком упрощенный дизайн затрудняет внедрение критически важной функции позже. |
| **Снижение когнитивной нагрузки:** Разработчику проще понять код, в котором нет «заготовок на будущее». | |

## Когда использовать (и когда НЕ использовать)

**Использовать:**
- В MVP (Minimum Viable Product) и Agile-процессах.
- В задачах с высокой неопределенностью требований.
- Когда стоимость поддержки лишнего кода выше, чем стоимость будущего рефакторинга.

**Не использовать (Исключения):**
- **Безопасность и масштабируемость:** Если архитектурное решение (например, выбор БД) крайне сложно заменить позже, основываясь на ожидаемом росте нагрузки, здесь YAGNI может быть опасен.
- **Стандарты отрасли:** Внедрение базового логирования или мониторинга с первого дня — это не «заготовка на будущее», а гигиенический минимум.

## Сравнение с альтернативами
YAGNI тесно связан с **KISS**. В то время как KISS говорит о *простоте реализации*, YAGNI говорит о *необходимости реализации* самой функции.

## Best Practices в .NET
1. **Рефакторинг вместо предугадывания:** Вместо того чтобы строить «гибкую систему», пишите простой код, который легко рефакторить.
2. **Используйте минимальные наборы зависимостей:** Не добавляйте NuGet-пакеты «на всякий случай».
3. **Фокус на текущем User Story:** Реализуйте ровно столько, сколько требуется для закрытия текущей задачи в Jira/Trello.
