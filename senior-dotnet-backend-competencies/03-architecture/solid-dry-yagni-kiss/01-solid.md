# SOLID: Фундаментальные принципы проектирования

## Теоретический фундамент
SOLID — это акроним пяти принципов объектно-ориентированного программирования, сформулированных Робертом Мартином. Их цель — создание гибкого, поддерживаемого и тестируемого кода, который легко расширять без внесения критических ошибок в существующий функционал.

---

## Разбор принципов и реализация в .NET

### 1. Single Responsibility Principle (SRP) — Принцип единственной ответственности
**Суть:** Класс должен иметь только одну причину для изменения. Это означает, что класс должен решать одну конкретную задачу.

- **Реализация в .NET:** 
    - Разделение бизнес-логики, доступа к данным (Repository) и логики представления (DTO/ViewModels).
    - Вместо одного «Божественного объекта» (`UserService`), создающего пользователей, отправляющего письма и логирующего действия, создаются отдельные сервисы: `UserRepository`, `EmailService`, `UserLogger`.
- **Нарушение:** Класс `OrderProcessor`, который считает сумму заказа, сохраняет его в БД и генерирует PDF-инвойс.

### 2. Open/Closed Principle (OCP) — Принцип открытости/закрытости
**Суть:** Программные сущности должны быть открыты для расширения, но закрыты для модификации.

- **Реализация в .NET:** 
    - Активное использование интерфейсов и абстрактных классов.
    - Использование паттерна «Стратегия» (Strategy). Например, если нужно добавить новый способ оплаты (PayPal, Stripe), мы добавляем новый класс `IPaymentMethod`, не меняя код основного сервиса оплаты.
- **Нарушение:** Использование больших конструкций `switch` или `if-else` для выбора поведения на основе типа объекта.

### 3. Liskov Substitution Principle (LSP) — Принцип подстановки Барбары Лисков
**Суть:** Объекты в программе должны быть заменяемы на экземпляры их подтипов без изменения корректности работы программы.

- **Реализация в .NET:** 
    - Наследник не должен «сужать» контракт базового класса (например, выбрасывать `NotImplementedException` для методов базового класса).
    - Если `Square` наследуется от `Rectangle`, но меняет логику установки ширины/высоты так, что нарушает инварианты `Rectangle`, это нарушение LSP.
- **Рекомендация:** Предпочитайте композицию наследованию, если подтип не является полноценной заменой базового типа.

### 4. Interface Segregation Principle (ISP) — Принцип разделения интерфейса
**Суть:** Клиенты не должны зависеть от методов, которые они не используют. Много специализированных интерфейсов лучше, чем один универсальный.

- **Реализация в .NET:** 
    - Разделение «толстых» интерфейсов на более мелкие. Вместо `IMachine { void Print(); void Scan(); void Fax(); }`, создать `IPrinter`, `IScanner`, `IFax`.
    - Это позволяет классам реализовывать только тот функционал, который им действительно нужен.
- **Нарушение:** Принуждение класса `BasicPrinter` реализовывать метод `Fax()`, который он не поддерживает.

### 5. Dependency Inversion Principle (DIP) — Принцип инверсии зависимостей
**Суть:** Модули высокого уровня не должны зависеть от модулей низкого уровня. Оба должны зависеть от абстракций.

- **Реализация в .NET:** 
    - Инъекция зависимостей (Dependency Injection).
    - Вместо `var service = new SqlRepository();` в контроллере, мы используем `public OrderController(IRepository repo)`.
    - Конкретная реализация подставляется контейнером (IServiceCollection) в `Program.cs`.

## Trade-offs (Плюсы и Минусы)

| Плюсы | Минусы |
| :--- | :--- |
| **Масштабируемость:** Легко добавлять новый функционал. | **Увеличение количества файлов:** Больше классов и интерфейсов. |
| **Тестируемость:** Легко создавать Mock-объекты для Unit-тестов. | **Порог входа:** Код может стать избыточным для простых задач (Overengineering). |
| **Снижение связанности:** Изменения в одном модуле меньше влияют на другие. | **Сложность навигации:** Нужно прыгать по интерфейсам, чтобы найти реализацию. |

## Когда использовать (и когда НЕ использовать)

**Использовать:**
- В долгосрочных коммерческих проектах.
- В сложных системах с частыми изменениями требований.
- Когда проект разрабатывается командой.

**Не использовать (или упрощать):**
- В маленьких утилитах, скриптах или MVP, где время выхода на рынок критичнее, чем идеальная архитектура.
- Если применение SOLID ведет к чрезмерному усложнению (Overengineering) простой логики.

## Сравнение с альтернативами
SOLID не противопоставляется другим подходам, а дополняет их. Например, в сочетании с **KISS** (Keep It Simple, Stupid) разработчик должен решать: «Будет ли применение DIP здесь оправдано или достаточно простого создания объекта?».

## Best Practices в .NET
1. **Интерфейсы для всего внешнего:** Любой внешний сервис (БД, API, Файловая система) должен быть скрыт за интерфейсом.
2. **Избегайте `NotImplementedException`:** Если вы реализуете интерфейс, но метод вам не нужен — значит, интерфейс нарушает ISP.
3. **Следите за размером классов:** Если класс занимает более 300-500 строк, скорее всего, он нарушает SRP.
