# Код-ревью (Code Review)

## Цели
- Поиск дефектов — ошибки, уязвимости, race condition.
- Передача знаний — команда узнаёт новые подходы.
- Единый стиль — соблюдение code style conventions.
- Архитектурный контроль — антипаттерны, преждевременные абстракции.

## Принципы конструктивного ревью
- Критикуем код, а не автора. "Здесь может быть null reference" vs "Ты снова забыл проверить null".
- Объяснять почему — "Лучше вынести в метод, потому что та же логика нужна в другом месте".
- Хвалить хорошие решения — "Отличное использование expression-bodied членов".
- Не мелочиться — code style может проверить линтер.

## Объём ревью
- **Small PR** (< 200–400 строк) — оптимально. Большие PR'a ревьюить сложнее и меньше дефектов находится.
- **Reviewed by 1-2 человека** — больше не даёт прироста качества.
- **Time to review** — чем быстрее, тем лучше (контекст не теряется).

## Чеклист
- Корректность (business logic).
- Безопасность (SQL injection, XSS, валидация).
- Производительность (N+1, неэффективные алгоритмы).
- Тестируемость (можно ли заюзать unit test).
- Readability (читаемость, naming, комментарии).

## Важно для интервью
- Gh-triggered review — процесс разработки через PR/Merge Request.
- Метод "стоп-линии" — больше N коммитов / больше M часов.
- PR шаблон с чеклистом — повышает качество ревью.
