# Технические спеки и RFC

## Tech Design Document (TDD)
- Документ, описывающий архитектурное решение до реализации.
- Содержит: проблему, предложение, альтернативы, аппроксимацию, план внедрения.
- Цель — собрать feedback и согласовать до начала разработки.

## ADR (Architecture Decision Record)
- Запись архитектурного решения с контекстом и обоснованием.
- **Title** — краткое название решения.
- **Context** — проблема и ограничения.
- **Decision** — что выбрали и почему.
- **Consequences** — последствия (хорошие и плохие).
- Статусы: Proposed → Accepted → Deprecated → Superseded.

## RFC (Request for Comments)
- Формат, заимствованный из Open Source (Python PEP, Rust RFC, Kubernetes KEP).
- Запрос на изменение/новую функциональность.
- Обсуждается и дорабатывается командой.
- В .NET — YARP, ASP.NET Core extensions.

## Best Practices
- **Краткость и ясность** — документ читают занятые люди.
- **Структура** — Context, Options, Recommendation, Timeline.
- **Visual** — диаграммы (C4, Mermaid, draw.io).
- **Хранить в репозитории** — docs/adr/ или docs/rfcs/.

## Важно для интервью
- Senior должен писать specs — это ожидаемый навык.
- ADR решает проблему "почему было принято такое решение" через полгода.
- Документ ≠ бюрократия — минимально достаточный объём.
