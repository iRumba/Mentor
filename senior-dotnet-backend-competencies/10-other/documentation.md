# Документирование кода и архитектуры

## Документирование кода
- **XML-комментарии** (C#) — `/// <summary>`, `<param>`, `<returns>`, `<example>`.
- **Не документировать очевидное** — комментарии нужны для "почему", а не "что".
- **Self-documenting code** — читаемые имена методов, классов (без комментариев).
- **Зачем комментарий**: бизнес-правило, workaround, performance hack, future intention.
- **Roslyn analyzers** — `SA1600` (все публичные элементы с XML-документацией).

## Документирование архитектуры
- **C4 Model** — контекст (System Context), контейнеры (Containers), компоненты (Components), код (Code).
- **Diagrams as Code** — PlantUML / Mermaid ([mermaid.live](https://mermaid.live)) — диаграммы в git.
- **ADR** (Architecture Decision Records) — см. топик 9.3.
- **README.md** — краткое описание проекта, как запустить, архитектура, ссылки.

## Best Practices
- **Живая документация** — автоматически генерируемая из кода (OpenAPI, ADR).
- **Wiki (Confluence, Notion)** — для общей документации, но может устаревать.
- **docs/ folder** — в репозитории, версионируется вместе с кодом.
- **Do not duplicate** — если информация в коде, не дублировать в wiki.

## Инструменты
- **DocFX** — генерация статического сайта документации из XML-комментариев.
- **Swashbuckle** — OpenAPI/Swagger UI.
- **Mermaid** — диаграммы как код (sequence, class, flow).

## Важно для интервью
- Документация = техника коммуникации + дисциплина.
- "Хорошая документация не та, которую легко писать, а та, которую легко читать".
- CI/CD может проверять устаревшую документацию (dead links).
