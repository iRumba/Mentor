# CQRS (Command Query Responsibility Segregation)

## Суть паттерна
- Разделение операций на **Command** (запись) и **Query** (чтение).
- Command — меняет состояние, не возвращает данные (void).
- Query — возвращает данные, не меняет состояние.

## Простой CQRS
- Одна БД, но разные модели для чтения и записи.
- Команды и запросы обрабатываются разными обработчиками.
- Меньше complexity, подходит для большинства проектов.

## Event Sourcing CQRS
- Команды порождают события, которые сохраняются в Event Store.
- Read модель строится из событий (projection).
- Разные БД для чтения и записи (гибридное хранение).

## Реализация на .NET
- **MediatR** — популярная библиотека для CQRS: `IRequest<T>` и `IRequestHandler<T>`.
- Команды/запросы — отдельные классы, обработчики — отдельные классы.
- Pipeline behaviours — cross-cutting concerns (логирование, валидация, транзакции).

## Важно для интервью
- CQRS разделяет **модели**, не просто методы.
- Когда оправдан (высокая нагрузка на чтение, сложная запись).
- Когда избыточен (простые CRUD).
- Отличие CQRS от Event Sourcing — первый не требует второго.
