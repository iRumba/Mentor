# CLR, GC, JIT, типы памяти (stack/heap)

## CLR (Common Language Runtime)
- Виртуальная машина .NET — управляет выполнением управляемого кода.
- Отвечает за JIT-компиляцию, GC, безопасность типов, исключения, потоки.
- Управляемый код компилируется в CIL (Common Intermediate Language), который JIT компилирует в нативный код при запуске.

## JIT (Just-In-Time)
- Компилирует IL-код в машинный код непосредственно перед выполнением метода.
- **Виды JIT**: 
  - *Tiered JIT* (с .NET Core 3.0) — сначала быстрая компиляция без оптимизаций (Tier 0), затем при повторных вызовах — оптимизированная (Tier 1).
  - *ReadyToRun (R2R)* — предварительная компиляция в нативный код при сборке.
  - *Native AOT* — полностью нативная компиляция без JIT (.NET 7+).

## GC (Garbage Collector)
- Автоматическое управление памятью: освобождает объекты, на которые нет ссылок.
- **Поколения**:
  - *Gen 0* — мелкие объекты, собираются часто.
  - *Gen 1* — пережившие Gen 0, собираются реже.
  - *Gen 2* — все остальные, собираются редко (full GC).
  - *Large Object Heap (LOH)* — объекты > 85 КБ, сразу в Gen 2.
- **Режимы GC**:
  - *Workstation GC* — для клиентских приложений.
  - *Server GC* — для серверов, несколько куч под каждый логический процессор.
  - *Background GC* — позволяет приложению работать во время сборки Gen 2.
- **Типы сборок**: blocking (приостанавливает потоки) / non-blocking (фоновая).

## Stack vs Heap
- **Stack** — хранит: value types (если не захвачены замыканием и не часть reference type), указатели на объекты в heap, аргументы метода, локальные переменные. LIFO, быстрый, потокобезопасный (свой у каждого потока).
- **Heap** — хранит: reference types, объекты. Управляется GC, медленнее, общий для всех потоков.

## Важно для интервью
- Разница между boxing (value → object, heap) и unboxing (object → value, stack).
- GC не детерминирован — `Dispose()` / `using` для освобождения ресурсов.
- Finalizer vs `IDisposable` vs `SafeHandle`.
- WeakReference, GCHandle, pinned objects.
